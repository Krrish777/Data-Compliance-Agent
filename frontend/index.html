    II.

    How a scan is conducted.

    A three-act rite: a policy is read, the data is examined, a report is impressed. Each act is recorded in a trace that cannot be rewritten.

    Act the First

    The Policy is Read.

    AuditLens ingests any regulatory PDF — HIPAA, GDPR, AML directives — and uses a Llama 3.3 70B model via Groq to extract structured compliance rules with confidence scores. Low-confidence rules surface immediately for human review before a single query is run.

    rule_extraction → human_review (if confidence < 0.7)
    Act the Second

    The Data is Scanned.

    Keyset-paginated queries traverse your tables without offset drift. Semantic PII detection identifies sensitive columns via sentence-transformer embeddings. A Python-side evaluator handles complex predicates — BETWEEN, regex, cross-field, date arithmetic — that resist round-tripping through plain SQL.

    schema_discovery → data_scanning → violation_validator
    Act the Third

    The Report is Issued.

    Every candidate violation is binary-classified by a fast Llama 3.1 8B validator. Confirmed violations receive a plain-language explanation from the 70B model. Results are rendered as a PDF and an HTML audit report, and written to an independent violations.db for independent querying.

    explanation_generator → report_generation → END

    III.

    The capabilities.

    Six instruments in the auditor's case, each purpose-built for a stage of the compliance rite.

    ⚖

    Dual-Model Strategy

    Rule extraction and violation explanation run on Llama 3.3 70B for maximum accuracy. Validation classification uses Llama 3.1 8B — faster and cheaper — keeping the critical path lean without sacrificing rule quality. Each model is pinned to the nodes where its cost-accuracy tradeoff fits best.

    🔒

    SQL Interceptor

    A second graph pre-screens individual SQL queries before execution. The interceptor classifies intent, maps it against policy embeddings stored in Qdrant, and returns an APPROVE or BLOCK verdict with a full audit trail — in real time.

    👁

    Human Review Gate

    Rules extracted with confidence below 0.7 are held at a LangGraph interrupt() checkpoint. A structured resume payload — {approved, edited, dropped} — lets a compliance officer correct, approve, or discard each rule before the scan proceeds.

    📄

    PDF & HTML Reports

    Every scan concludes with a timestamped audit report in both PDF and HTML. Reports include violation counts by severity, affected tables, rule citations, and LLM-generated remediation guidance — ready to hand to a regulator. The full trace is independently queryable via violations.db.

    📊

    Independent Violation Store

    Violations are persisted to a separate violations.db SQLite file, not bundled inside LangGraph checkpoints. The audit trail is independently queryable, survives graph restarts, and stays small enough to version-control alongside your schema.

    🔗

    Vector Policy Memory

    Policy documents are embedded with BAAI/bge-small-en-v1.5 via FastEmbed and stored in a local Qdrant instance. Semantic similarity powers both the interceptor's policy lookup and long-term cross-thread learning of prior rule corrections.
